# Layered Solver Prototype

The full reachable graph is too large to keep as one in-memory graph. A 100M CSR no-predecessor probe was still truncated, only reached the 7-soldier layer, and had not reached layers 6..0. The solver route is therefore moving toward layered reachability, layered retrograde, and compressed or externalized tables.

## Soldier-Count Monotonicity

The game has a useful monotone structure:

- normal cannon moves keep the soldier count unchanged;
- soldier moves keep the soldier count unchanged;
- cannon captures reduce the soldier count by one;
- no legal move increases the soldier count.

Reachability can therefore be generated from high layers to low layers:

```text
15 soldiers -> 14 soldiers -> ... -> 0 soldiers
```

Later retrograde solving will work in the opposite direction, using solved lower layers to help solve higher layers. This prototype only builds reachable state files; it does not solve outcomes.

## Current Prototype

`buildReachableLayers(...)` starts from the standard initial position. `buildReachableLayersFromSeed(...)` exists for small tests and future custom-position tooling.

For each soldier count `k`, the builder:

1. reads `seeds-k.s15seed`;
2. computes the same-layer closure for states that still have `k` soldiers;
3. records capture successors as seeds for layer `k - 1`;
4. sorts and deduplicates layer states;
5. writes `layer-kk.s15layer`;
6. writes `seeds-(k-1).s15seed` when `k > 0`.

The initial standard build writes `seeds-15.s15seed` automatically. Resumed builds start from an existing `seeds-k.s15seed`; they do not assume the previous layer was complete.

The default `--build-layers` path still uses `std::unordered_set<uint64_t>` and `std::queue<uint64_t>` for layer-local closure. Passing `--external-layer-closure` switches each layer to the sorted-file closure backend. Capture-generated next-layer seed dedup can use external sorted runs with `--external-seed-dedup`.

## Layer File Format

Layer files use a small binary v1 format:

```text
magic[8]      "S15LYR1\0"
uint32_le     version = 1
uint32_le     soldierCount
uint64_le     stateCount
uint64_le     rulesetHash
uint64_le[]   sorted state keys
```

The v1 file stores only state keys. It does not store edges, outcomes, distances, flags, or truncation status. If a build is truncated, the CLI and stats report that fact; the layer file itself is just a key set.

The reader validates magic, version, ruleset hash, expected soldier count, strict key ordering, uniqueness, and each key's decoded soldier count.

## Seed File Format

Seed files use the same key-list layout, with a different magic:

```text
magic[8]      "S15SED1\0"
uint32_le     version = 1
uint32_le     soldierCount
uint64_le     seedCount
uint64_le     rulesetHash
uint64_le[]   sorted seed keys
```

`.s15layer` files contain reachable states for one layer. `.s15seed` files contain entry points generated by captures from the previous layer. Persisting seeds makes large builds resumable one layer at a time.

## Manifest

Each `--build-layers` run writes `manifest.json` with the build status, start/stop layers, resumed flag, closure backend, truncation reason, per-layer counts, file names, skipped-layer metadata, external closure checkpoint/resume fields, external closure timing, and external seed-dedup statistics when enabled. The manifest is helper metadata; readers still validate `.s15layer`, `.s15seed`, and checkpoint key files directly.

## CLI

```powershell
.\build\sanpao15_cli.exe --build-layers build\layers --max-states-per-layer 1000000 --progress 50000
```

`--build-layers DIR` writes layer and seed files as far as the build can safely go. It does not run retrograde and does not write `.s15tbl`.

`--max-states-per-layer N` caps each layer. `0` means unlimited. The builder stops after a truncated layer so later files are not mistaken for complete downstream layers.

Use external sorted runs for capture seed dedup:

```powershell
.\build\sanpao15_cli.exe --build-layers build\layers-ext --stop-after-layer 15 --max-states-per-layer 1000000 --external-seed-dedup --dedup-chunk-size 100000 --progress 50000
```

`--external-seed-dedup` leaves layer-local visited membership in memory, but replaces the next-layer seed `unordered_set` with chunked sorted runs and k-way merge. `--dedup-chunk-size N` controls the number of keys per temporary run. `--temp-dir DIR` chooses the root temp directory; otherwise runs are written below `<outputDir>/tmp/layer-K`. `--keep-temp` keeps `.s15run` files for inspection.

Build one layer with the external closure prototype:

```powershell
.\build\sanpao15_cli.exe --build-layer-external 15 --layer-work-dir build\layer15-work --seed-file build\layers\seeds-15.s15seed --output-layer build\layer15-work\layer-15.s15layer --output-next-seed build\layer15-work\seeds-14.s15seed --dedup-chunk-size 1000 --max-expanded-states 10000 --progress 1000
```

This uses internal `.s15keys` files for `visited`, `frontier`, `candidates`, and set operations. It writes normal `.s15layer` and `.s15seed` outputs.

Use external closure in the layered builder:

```powershell
.\build\sanpao15_cli.exe --build-layers build\layers-ext --external-layer-closure --external-seed-dedup --stop-after-layer 15 --max-expanded-states 1000000 --dedup-chunk-size 100000 --progress 50000
```

`--max-states-per-layer` limits discovered/final states. `--max-expanded-states` limits expanded frontier states. In truncated external runs, discovered states can be larger than expanded states because the next frontier has already been found.

External closure writes a checkpoint under `<outputDir>\work\layer-K` by
default. A second slice can resume the same layer:

```powershell
.\build\sanpao15_cli.exe --build-layers build\layers-ext --external-layer-closure --external-seed-dedup --resume-closure --start-layer 15 --stop-after-layer 15 --max-expanded-states 1000000 --dedup-chunk-size 100000 --progress 50000
```

The checkpoint contains `closure-state.json`, `visited.s15keys`,
`frontier.s15keys`, `next-seeds.s15keys`, and, when a slice stops mid-iteration,
`base-visited.s15keys` plus `pending-candidates.s15keys`. These files preserve
the same iteration boundary as an uninterrupted external closure run. In resume
mode, `--max-expanded-states N` is an additional budget for the current command.
`--checkpoint-interval N` writes intermediate checkpoints every roughly `N`
additional expanded states; `0` means completion/truncation only.

Checkpoint manifests are hardened so resume depends only on stable materialized
`.s15keys` files. Version 2 records `checkpointKind`,
`requiresTransientRuns=false`, and `remainingFrontierFile`. Stale temporary
`.s15run` directories are ignored by validation and can be removed through the
repair/cleanup path after the stable checkpoint validates.

Manifest layer entries record `resumedClosure`, `complete`,
`checkpointWritten`, `checkpointDir`, `initialVisitedStates`,
`initialFrontierStates`, and `finalFrontierStates`. `--validate-layers` validates
checkpoint manifests and their referenced `.s15keys` files when present. A
truncated checkpoint is only a resumable partial layer; it is not a complete
layer.

Partitioned checkpoint snapshots can be enabled for small smoke runs and future
migration work:

```powershell
.\build\sanpao15_cli.exe --build-layers build\partitioned-closure-smoke --external-layer-closure --partitioned-closure --external-seed-dedup --stop-after-layer 15 --max-expanded-states 10000 --closure-partition-buckets 16 --dedup-chunk-size 1000 --progress 1000
```

This v1 foundation still runs the flat external closure algorithm, but writes
partitioned checkpoint directories for `visited`, `frontier`,
`remaining-frontier`, and `next-seeds`. When a mid-iteration checkpoint has
pending work, it also partitions `pending-candidates` and `base-visited`.
`--validate-layers` validates these partitioned snapshots when the checkpoint
manifest records `partitionedClosure=true`.

Build only layer 15 and persist the next seed file:

```powershell
.\build\sanpao15_cli.exe --build-layers build\layers --stop-after-layer 15 --progress 50000
```

Resume from an existing seed file:

```powershell
.\build\sanpao15_cli.exe --build-layers build\layers --start-layer 14 --stop-after-layer 14 --progress 50000
```

Validate or inspect files:

```powershell
.\build\sanpao15_cli.exe --validate-layers build\layers
.\build\sanpao15_cli.exe --inspect-layer build\layers\layer-15.s15layer
.\build\sanpao15_cli.exe --inspect-seed build\layers\seeds-14.s15seed
```

Build a partitioned membership index from a large layer or seed file:

```powershell
.\build\sanpao15_cli.exe --partition-keyset build\layers-overnight\layer-15.s15layer --partition-output build\partitions\layer-15 --partition-buckets 256 --progress 1000000
.\build\sanpao15_cli.exe --validate-partition build\partitions\layer-15
.\build\sanpao15_cli.exe --inspect-partition build\partitions\layer-15
.\build\sanpao15_cli.exe --benchmark-partition build\partitions\layer-15 --sample 100000
```

The partitioned keyset is still reachability infrastructure. It does not solve outcomes, but it gives later layer-local edge generation a `key -> bucket -> binary search` membership path.

New partitions default to `splitmix64_mod`; old `key_mod` partitions remain readable through the manifest's `partitionMethod`.

Layer-local edge probe:

```powershell
.\build\sanpao15_cli.exe --probe-layer-edges build\partitions-splitmix\layer-15 --next-seed-partition build\partitions-splitmix\seeds-14 --sample-states 100000 --partition-cache-buckets 32 --progress 10000
```

This expands sampled states and checks partition membership for same-layer and capture successors. It does not store edges and does not run retrograde. On partial layer data, missing successor counts are trend/performance observations, not proof of invalid data.
